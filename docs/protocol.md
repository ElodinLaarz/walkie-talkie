# Frequency Wire Protocol — v1

This document defines the wire protocol that Frequency phones use to discover
each other, join a "frequency", and coordinate voice + shared-media state. It
is the contract that subsequent implementation PRs (real BT scanning, voice
transmission, media sync) build against.

## Goals

- **Bluetooth-only.** No internet required for voice (per the README and the
  app's pitch). Same goes for control: discovery, join, mute, media commands.
- **Android-first**, targeting API 33+ (LE Audio).
- **Star topology** with a designated host. Voice and control traffic both go
  through the host, who is the coordinator for the frequency. Mesh and host
  handover are out of scope for v1 (see [§ Out of scope](#out-of-scope)).
- **Two planes**: a **control plane** carried over Bluetooth LE GATT in JSON,
  and an **audio plane** carried over Bluetooth LE Audio CIS / BIS at the
  native layer. This document specifies the control plane only.

## Topology

```text
                 ┌───────────────────────┐
                 │       HOST phone      │
        ┌────────┤  • LE advertiser      ├────────┐
        │        │  • GATT server        │        │
        │        │  • LE Audio source    │        │
        │        └───────────────────────┘        │
        │                    ▲                    │
        │ control            │ control            │ control
        │                    │                    │
        ▼                    ▼                    ▼
 ┌───────────┐        ┌───────────┐        ┌───────────┐
 │ guest #1  │        │ guest #2  │        │ guest #3  │
 └───────────┘        └───────────┘        └───────────┘
```

Every guest holds exactly one GATT connection: to the host. Guests do not talk
to each other directly. The host re-broadcasts roster updates, talking state,
mute state, and media commands.

**Why star?**

- Bluetooth LE phones typically support 4–7 simultaneous connections; a full
  mesh is N² and saturates fast.
- Single mixer in the audio plane (the "mix-minus" matrix the README
  describes) lives naturally on one device — the host.
- Far simpler reliability story for v1. Host leaves → frequency ends.

## Identifiers

| name           | shape                 | how it's made                                   | lifetime               |
| -------------- | --------------------- | ----------------------------------------------- | ---------------------- |
| `peerId`       | UUID v4 string        | generated once per install, persisted in Hive   | install lifetime       |
| `sessionUuid`  | UUID v4 string        | generated by host on *Start a new Frequency*    | room lifetime          |
| `sessionCode`  | 4-char base32         | derived deterministically from `sessionUuid`    | room lifetime          |
| `mhzDisplay`   | string like `"104.3"` | derived deterministically from `sessionUuid`    | room lifetime, cosmetic |

The MHz value is **purely cosmetic** — it's what the UI shows on chrome
("On air · 104.3"), invite sheets, and discovery rows ("On 104.3"). Routing
is always by `sessionUuid`. Two simultaneous sessions can collide on the same
MHz string; guests disambiguate by host name + RSSI.

The deterministic mapping uses the low 12 bits of `sessionUuid`, mod 200:

```text
tenths = 880 + (low_12_bits % 200)
mhz    = tenths / 10.0
```

This produces 200 evenly-distributed buckets in the real-world FM band
[88.0, 107.9] MHz at 0.1-MHz precision.

The deterministic mapping for `sessionCode`, since it's user-facing and must
match across platforms:

1. Take the canonical UUID string with hyphens stripped (hex parsing is
   case-insensitive, so callers may use either case).
2. Take the **last 5 hex characters** and parse as an unsigned 20-bit integer
   (big-endian — the first of those 5 chars is the high nibble).
3. Encode as **4 Crockford-base32 digits**, **least-significant-5-bits first**:
   digit 0 is bits 0–4 of the integer, digit 1 is bits 5–9, digit 2 is bits
   10–14, digit 3 is bits 15–19.
4. Alphabet is `0123456789ABCDEFGHJKMNPQRSTVWXYZ` (32 chars; excludes `I`,
   `L`, `O`, `U` to avoid look-alike characters).

## Bluetooth LE advertising

The host advertises a single 128-bit service UUID and 16 bytes of manufacturer
data:

| offset | bytes | meaning                                  |
| ------ | ----- | ---------------------------------------- |
| 0      | 1     | protocol version (`0x01` for v1)         |
| 1      | 1     | role (`0x01` = host)                     |
| 2      | 8     | low 8 bytes of `sessionUuid`             |
| 10     | 2     | flags (reserved, zero in v1)             |
| 12     | 4     | reserved                                 |

Guests scan for the service UUID, parse the manufacturer payload, and render
the result in the Discovery list. The advertised local name is the host's
display name, truncated to fit the LE adv frame.

Encrypted / authenticated advertising is out of scope for v1; LE pairing
handles link-layer crypto and the app trusts that boundary.

## GATT service

Service UUID: `WALKIE_TALKIE_SERVICE` — the same 128-bit UUID used in
advertising.

Two characteristics:

| name      | properties           | direction      | purpose                              |
| --------- | -------------------- | -------------- | ------------------------------------ |
| REQUEST   | write, write-no-resp | guest → host   | guest sends control messages         |
| RESPONSE  | notify               | host → guest   | host pushes events to subscribers    |

Each characteristic carries a single line-delimited JSON document per write or
notification. Default GATT MTU is 23 bytes; v1 negotiates up to 247 bytes (the
practical Android max). A message that exceeds the negotiated MTU is split
into 247-byte fragments. Each fragment is prefixed with a **4-byte header**:

| offset | bytes | meaning                                                        |
| ------ | ----- | -------------------------------------------------------------- |
| 0      | 1     | `0x00` — fixed in v1, reserved for a future fragment-flags byte |
| 1–2    | 2     | `total_len` — total JSON length in bytes, **big-endian uint16** |
| 3      | 1     | `fragment_idx` — zero-based index of this fragment              |

That leaves up to 243 bytes of JSON payload per fragment:

```text
[ 0x00  total_len_hi  total_len_lo  fragment_idx ]  <up-to-243-bytes-of-JSON>
```

In v1 we cap each message at **4 KiB** (≈ 17 fragments). The roster update at
the design's max group size of 12 peers can reach ~2 KiB once display names and
BT-device strings are included; 4 KiB leaves a comfortable margin without
forcing length-prefixed reframing yet.

Connection MTU negotiation is the responsibility of the platform layer; the
control plane just sees byte streams to write and notifications to consume.

## Message envelope

Every control-plane message is a JSON object with at minimum:

```json
{
  "kind": "<message kind>",
  "peerId": "<sender peerId>",
  "seq": <monotonic int starting at 1>,
  "atMs": <sender's ms-since-epoch>,
  "v": 1
}
```

- `kind` selects the schema for the rest of the body.
- `seq` is a per-sender monotonically increasing int. Receivers track the last
  `seq` seen per peer and drop out-of-order or duplicate messages.
- `atMs` is the sender's wall-clock at send time. Used for "12s ago" display
  and for ordering media commands when seqs come from different peers. We
  expect drift up to ±5s between peers; that's acceptable for UI purposes.
- `v` is the protocol version. A v1 parser **MUST** reject messages with
  `v != 1` (Dart implementation throws `FormatException`); the receiver loop
  catches that, drops the message, and keeps the connection up. Net effect:
  v != 1 messages are ignored, no further processing.

## Message catalogue

### Lifecycle

#### `join_request` (guest → host)

Sent by a guest right after the GATT connection is up.

```json
{
  "kind": "join_request", "peerId": "...", "seq": 1, "atMs": 1714060800000, "v": 1,
  "displayName": "Maya",
  "btDevice": "AirPods Pro"
}
```

Host responds with either `join_accepted` or `join_denied` over RESPONSE.
Guest **MUST** wait for the response before sending any further messages.

#### `join_accepted` (host → guest)

```json
{
  "kind": "join_accepted", "peerId": "<host>", "seq": 7, "atMs": ..., "v": 1,
  "hostPeerId": "<host>",
  "roster": [ { "peerId": "...", "displayName": "Maya", "btDevice": "AirPods Pro",
                "muted": false, "talking": false }, ... ],
  "mediaState": { "source": "YouTube Music", "trackIdx": 2,
                  "playing": true, "positionMs": 37000 }
}
```

The full roster (including the new guest) and the current media state. Guest
uses these to render the room screen.

#### `join_denied` (host → guest)

```json
{ "kind": "join_denied", "peerId": "<host>", "seq": ..., "atMs": ..., "v": 1,
  "reason": "host_declined" }
```

Reasons: `host_declined`, `room_full`, `version_mismatch`. Guest disconnects
on receipt.

#### `leave` (guest → host) and `remove_peer` (host → all)

```json
{ "kind": "leave", "peerId": "<sender>", "seq": ..., "atMs": ..., "v": 1 }
```

Guest sends `leave` and disconnects. Host emits a follow-up `roster_update`
to remaining guests.

```json
{ "kind": "remove_peer", "peerId": "<host>", "seq": ..., "atMs": ..., "v": 1,
  "target": "<peerId being kicked>" }
```

Host writes `remove_peer` to RESPONSE; the targeted peer disconnects on
receipt and other guests update their roster.

#### `roster_update` (host → all)

```json
{ "kind": "roster_update", "peerId": "<host>", "seq": ..., "atMs": ..., "v": 1,
  "roster": [ ... ] }
```

Pushed whenever the roster changes (join, leave, removal, mute toggle).

### Voice control

These do **not** carry audio — they describe state for UI rendering. Audio
itself flows on a separate plane (LE Audio CIS / BIS).

#### `talking` (peer → host → all)

```json
{ "kind": "talking", "peerId": "<sender>", "seq": ..., "atMs": ..., "v": 1,
  "talking": true }
```

Sent when the peer's mic crosses the voice-activity threshold. Host
re-broadcasts. Drives the talking-ring + VU meter on every peer's screen.

#### `mute` (peer → host → all)

```json
{ "kind": "mute", "peerId": "<sender>", "seq": ..., "atMs": ..., "v": 1,
  "muted": true }
```

Self-mute or unmute. Host re-broadcasts.

### Shared media

Per the README's DRM section we cannot capture/restream YouTube/Spotify — only
**control signals** propagate. Each peer plays its own copy of the media; the
host is the source of truth for what's playing and where.

#### `media` (any peer → host → all)

```json
{ "kind": "media", "peerId": "<sender>", "seq": ..., "atMs": ..., "v": 1,
  "op": "play"|"pause"|"skip"|"prev"|"seek"|"queue_play",
  "source": "YouTube Music",
  "trackIdx": 3,
  "positionMs": 91500
}
```

`trackIdx` is required for `queue_play`, optional for `skip`/`prev`/`seek`.
`positionMs` is required for `seek`, ignored otherwise.

Host applies the command, updates its mediaState, then re-broadcasts. The
broadcast is what every peer (including the originator) consumes — the
originator's local optimistic update is reconciled against the host's echo.

### Health

#### `signal_report` (peer → host)

```json
{ "kind": "signal_report", "peerId": "<sender>", "seq": ..., "atMs": ..., "v": 1,
  "neighbors": [ { "peerId": "...", "rssi": -71 } ] }
```

Sent periodically (~10s). Host uses it to detect weak links and surface the
"Maya's signal is weak" toast.

#### `ping` (peer → host)

```json
{ "kind": "ping", "peerId": "<sender>", "seq": ..., "atMs": ..., "v": 1 }
```

Sent every ~5s. Host considers a peer dropped after 3 missed pings.

## Walk-through: host advertise → guest tune in

```text
HOST                                          GUEST
────                                          ─────
tap "Start a new Frequency"
generate sessionUuid
derive sessionCode + mhzDisplay
start LE advertising (service UUID + payload)
start GATT server (REQUEST/RESPONSE chars)
render Room screen
                                              tap Discovery / start scan
                              ◄────── LE adv (sessionUuid, role=host, v=1)
                                              parse adv, derive mhz, render row
                                              tap "Tune in"
                              ◄────── GATT connect
                              ◄────── subscribe to RESPONSE
                              ◄────── write JoinRequest (REQUEST)
host receives JoinRequest
emits "wants to tune in" toast
user taps Let in
                              ──────► JoinAccepted (RESPONSE notify)
                                              render Room screen
                              ──────► RosterUpdate (broadcast)

(later — anyone scrubs media)
                              ◄────── MediaCommand seek 91500ms (REQUEST)
update mediaState
                              ──────► MediaCommand seek 91500ms (RESPONSE notify, all)

(later — guest leaves)
                              ◄────── Leave (REQUEST)
                              ◄────── GATT disconnect
                              ──────► RosterUpdate (RESPONSE notify)
```

Voice is brought up *after* `JoinAccepted` by the native LE Audio layer; the
control plane carries the announcement that voice is now expected.

## Sequence numbers and idempotency

- Each peer keeps a monotonic `seq` counter starting at 1, incremented on
  every send (regardless of recipient).
- Receivers store `lastSeq[peerId]` and drop messages where
  `msg.seq <= lastSeq[peerId]`.
- After reconnect, peers reset their counter to 1 and receivers clear their
  table for that peerId.

This handles BLE's at-least-once write semantics without app-level acks.

## Error handling

- **Unknown `kind`** — log, drop. Forward-compatible: a v2 host can send v2
  message kinds to v2 guests; v1 guests will ignore them.
- **Malformed JSON** — log, drop. Disconnect after 3 consecutive malformed
  messages from the same peer.
- **Version mismatch** — host sends `join_denied { reason: "version_mismatch" }`
  if the guest's adv-or-message `v` differs from its own.

## Out of scope

- **Voice transport.** LE Audio CIS / BIS setup is native (Kotlin). The
  control plane only signals when voice is expected.
- **Encryption / authentication.** Link-layer LE pairing is enough for v1.
  App-level signing of messages is a v2 concern.
- **Host handover.** v1: host leaves → channel ends, all guests return to
  Discovery. v2 will pick the longest-connected guest as new host.
- **Mix-minus.** The host's responsibility, native side. Each guest's voice
  packet is broadcast to all *other* guests; the originator hears only what
  others say.
- **Beyond 12 peers.** The roster/media JSON envelope assumes ≤12 peers
  (the design's max group size). Larger groups need a different framing.

## Versioning

This document is **v1**. The `v: 1` field on every message and the
`version: 0x01` byte in advertising data identify the wire format.

A v2 protocol must:

1. Bump the version byte in advertising, so v1 scanners don't try to join.
2. Either keep `v=1` messages decodable for graceful overlap, or set
   `version_mismatch` on `join_denied`.
